# Contributing to Vibe Coding Guide

Thanks for your interest in contributing! This guide is built on real experience, and the best contributions come from practitioners who've discovered patterns that work.

## What We're Looking For

**Great contributions:**
- New patterns you've battle-tested across multiple projects
- Improvements to existing guides with real-world examples
- Templates that make adoption easier
- Translations (especially for non-English developer communities)
- Corrections to advice that didn't hold up in practice

**Not a good fit:**
- Theoretical patterns you haven't personally used
- Framework-specific advice (this guide is framework-agnostic)
- Promotional content for specific tools or services
- Minor wording tweaks that don't change meaning

## How to Contribute

1. **Open an issue first** — Describe the pattern or improvement. Include context on where you learned it and why it matters.
2. **Fork and create a branch** — Use a descriptive branch name like `add-error-handling-pattern` or `improve-tdd-examples`.
3. **Write your contribution** — Follow the style guide below.
4. **Submit a PR** — Reference the issue. Include a brief summary of what you're adding and why.

## Style Guide

### Voice
- Write in second person ("you") — direct and actionable
- Be opinionated — "do this" beats "you could consider doing this"
- Include the "why" — every rule needs a reason
- Be honest about trade-offs — when does a pattern NOT apply?

### Structure
Each guide should follow this pattern:
1. **What** — The pattern in one sentence
2. **Why** — The problem it solves (ideally from a real failure)
3. **How** — Step-by-step implementation
4. **Anti-patterns** — What NOT to do
5. **When to skip** — Honesty about when this doesn't apply

### Examples
- Use multi-language examples where possible (don't assume TypeScript)
- Show both good and bad examples side by side
- Keep code snippets short and focused

### Formatting
- Use GitHub-flavored markdown
- Keep line lengths reasonable (no hard wrap requirement)
- Use headers for navigation (H2 for sections, H3 for subsections)
- Tables for comparisons, bullet lists for sequences

## Code of Conduct

Be respectful, constructive, and focused on helping developers build better with AI. This is a technical resource — keep discussions technical.
